# 搜索研究 Agent 设计困境——请求外部建议

## 背景

我正在构建一个**基于免费版大模型网页端**的搜索研究 Agent。它不是调 API，而是通过 CDP 协议操控浏览器里的 AI 对话平台（DeepSeek、Kimi、ChatGPT、Gemini、SCnet 等），自动发送搜索 prompt 并提取结构化回复。

### 运行环境
- Windows 11，Python 3.12，Playwright（CDP 操控 Edge 浏览器）
- 所有 AI 平台均使用**免费版网页端**（非 API），登录态通过 CDP 浏览器保持
- 提取机制：在 prompt 中嵌入 `[搜索主题：xxx@hash]` 标记对，用标记定位 AI 回复的正式内容区域

### 已实现的能力
- **平台定型**：自动分析每个平台的 DOM 结构，生成交互脚本（填输入框、点发送、等回复）
- **内容自进化**：每平台独立的提取档案（CoT 关键词、标记位置模式、页尾特征），失败时自动学习并持久化
- **多平台测试状态**：SCnet（DeepSeek V4 Pro 满血版）✓、ChatGPT 免费版 ✓、DeepSeek 免费版 ✓、Kimi（DOM 已适配，登录态待确认）

## 当前架构：一条问题的完整旅程

```
用户问题
  ↓
① 工具选择层（ToolRouter）
   分类：WebSearch（单点事实）还是 WebAISearch（需要分析推理）
   分级：L1 快速确认 / L2 标准调研 / L3 深度研究
   方式：deepseek-v4-flash 做语义分类（原为纯规则关键词匹配，已被否掉）
  ↓
② L1 → 不走 Agent（直接 WebSearch 工具），L2/L3 → 进入下方流程
  ↓
③ 规划层（Planner）
   将问题分解为子问题（≤5 个），每个子问题路由到最优平台
   方式：deepseek-v4-pro 做分解 + 路由（输出 Q:xxx | ROUTE:xxx）
   路由选项：moderate_code→DeepSeek / complex_arch→Kimi / english_search→Gemini /
             simple_code_en→Perplexity / chinese_academic→秘塔 / final_report→Kimi
  ↓
④ Prompt 生成（PromptBuilder）
   每个子问题生成搜索 prompt（含搜索规则 + 输出格式 + 标记对）
   三种模式：code（技术选型/方案对比）/ debug（Bug排查）/ news（新闻热点）
   每种模式 × 三个深度（L1/L2/L3）= 9 套 prompt 模板
  ↓
⑤ 执行编排层（Orchestrator）
   依次为每个子问题：打开平台页面 → 填 prompt → 发送 → 轮询等待 → 提取回复
   含自适应轮询（PollingProfile）、CoT 检测、强制提取兜底
  ↓
⑥ 整合层（Synthesizer）
   多平台结果 → 交叉验证 → 信源评分 → 矛盾检测 → 生成最终研究报告
   （当前仅基础版本，无 judge 评分机制）
  ↓
最终报告
```

## 核心困境

### 困境 1：Planer 拆分 vs Prompt 内嵌维度——两套分解机制在打架

**Planer 拆分（步骤③）**：把一个大问题拆成多个子问题，每个子问题发给不同平台。例如"Rust 为什么成为最受喜爱的语言"拆成：
- 子问题 1 → Gemini：调查数据
- 子问题 2 → Kimi：核心优势分析
- 子问题 3 → Kimi：应用场景分析

**Prompt 内嵌维度（步骤④）**：每个子问题的 prompt 里已经包含了多维分析指令。例如 code 模式的 prompt 要求回复覆盖"性能、兼容性、学习成本、生态、社区活跃度"五维对比，输出 7 段结构（概述→核心要点→方案对比表→可操作性评估→风险评估→来源评价→局限性说明）。

**问题**：Planer 拆出来的子问题（"Rust 的核心优势是什么"）和 prompt 内嵌维度（"从五个维度对比分析"）之间是什么关系？它们是互补还是冲突？如果 prompt 已经让模型按五维分析了，Planer 再拆成三个子问题发给不同平台，是增加了交叉验证的价值，还是造成了信息碎片化？

### 困境 2：平台路由全都走 Kimi——LLM 对平台能力的理解不准确

当前 Planner 的 LLM（deepseek-v4-pro）做分解时，几乎所有问题都路由到 Kimi。但实际上：
- SCnet（超算平台）使用 DeepSeek V4 Pro 满血版，支持文件上传（关联网搜索即可上传），搜索限制约 10 个网页，天然适合做最终报告整合
- DeepSeek 免费版当前仅快速模式可用，分析能力受限
- Kimi 虽然中文信息覆盖最强（WideSearch 80.8%），但**速度最慢**（可能需要独立轮询策略）
- ChatGPT 免费版严重阉割且限流

这些实时状态 LLM 不知道，也不可能知道——它们是变化的（上周 DeepSeek 还有 DeepThink，这周可能没了）。

### 困境 3：拆不拆、拆几个——L3 深度研究强制分解失效

代码里 L3 会强制触发分解（`force_decompose=True`），但 LLM 经常返回只有 1 个子问题。例如"从零构建企业级 LLM 推理平台的全面技术评估"（L3）→ 返回了 1 个子问题。导致 L3 特有的 final_report 整合步骤（需要 `子问题 > 1` 才触发）直接跳过。

是 prompt 不够强硬，还是这类问题本身就适合作为一个整体发给模型（而非强行拆分）？

### 困境 4：没有 judge 层——拆得好不好、搜得对不对，无从判断

当前全流程没有评分反馈机制。三个自进化（配置/决策/内容）都需要一个裁判来提供"结果好不好"的信号，但这个裁判还不存在。在缺少反馈环的情况下，Planner 的拆分质量、平台路由的合理性、最终报告的完整性，全部靠人工抽查。

## 约束条件

1. **免费版网页端**：不能调 API（除了用于 Planner 路由分类的 deepseek-v4-flash/pro API），所有搜索走浏览器网页端
2. **平台行为不可控**：DOM 随时变、模型随时换、功能随时改（如 DeepSeek 快速模式限制）
3. **速度差异巨大**：Gemini Flash 最快（秒级），Kimi 最慢（分钟级），不能简单并行
4. **不能引入额外 LLM 调用链**：每多一次 LLM 调用就多一个故障点，设计上倾向于让负责最终整合的平台同时做评分
5. **标记对定位机制**：通过在 prompt 中嵌入标记来定位 AI 正式回复内容区，但不同模型对标记的处理不同（有些把标记放在思考块之前，有些放在之后）

## 希望得到的建议

1. **Planer 和 Prompt 的分解职责如何划分？** Planer 做"多平台分派"，Prompt 做"单平台多维分析"是否合理？还是应该放弃 Planer 分解，让每个平台用同一个完整 prompt 各自发挥，靠多平台结果做交叉验证？
2. **平台路由如何变得更智能？** 除了硬编码路由表 + 性能统计自动调整外，有没有更优雅的方式让系统自己学会"什么任务用什么平台"？
3. **Judge 层应该做多深？** 是否有比"让整合平台同时输出评分 JSON"更轻量的反馈机制？

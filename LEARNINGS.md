## 2. 架构决策记录（ADR）

### ADR-1: 为什么分两种"定型"

网页端和本地端虽然都遵循"一次搞定、缓存复用"，但机制完全不同：
- **平台定型**：LLM 分析 DOM → 生成 Playwright 交互代码（代码生成）
- **本地定型**：LLM 运行探测命令 → 解析输出 → 写 JSON（配置发现）

分开的理由：平台定型的结果是**可执行代码**（需 import 验证），本地定型的结果是**结构化数据**（需 JSON Schema 验证）。混在一起会导致类型混乱。

### ADR-2: 为什么用标记对而非 DOM 选择器做提取

DOM 选择器（如 .response-content、[data-message]）依赖平台 UI 实现，每个平台不同、每个版本可能变化。标记对 [搜索主题：xxx@hash] 是**我们注入到 prompt 中的文本**，AI 必须输出，完全平台无关。

代价：需要在 prompt 中增加定位要求（约 100 字符），但对搜索结果质量无影响。

### ADR-3: 为什么新鲜度阈值是 1 而非 50%

标记匹配已经精确到了 topic 级别（@hash 保证全局唯一）。新鲜度验证是**辅助检查**，防止极端情况（如标记因编码问题被截断）。阈值设 50% 会导致长 topic 文本在 AI 回复中必然失败（AI 不会复述问题原文）。

### ADR-4: 为什么 prompt 生成用 import 而非 shell 管道

PowerShell 管道传中文必损坏编码。python prompt_builder.py "中文" 在 Windows 上不可靠。改为 python -c "from prompt_builder import ..." 的方式，数据全程在 Python 内存中，不经过 shell。

### ADR-5: 为什么平台脚本放 scripts/platforms/ 而非 platforms/

历史原因：generator.py 在 scripts/ 下，PLATFORMS_DIR = os.path.join(SCRIPTS_DIR, "platforms")。如果改到根目录，需要同时改 generator.py、main.py 的 import 路径、SKILL.md 的文档。属于"可以但不值得"的重构。

## 4. 通用设计模式（可迁移到其他 skill）

### 模式 1: 标记对截取（Marker Pair Extraction）

**问题**：从混合了多轮对话的页面文本中精确提取某一次 AI 回复。

**方案**：
1. 在 prompt 中要求 AI 在回复首尾各加一个唯一标记
2. 标记含 hash 保证不同轮次不冲突
3. 用 find(prompt_text) 定位本次发送
4. 在"本次发送之后"的区域收集标记
5. 取相邻标记对截取内容

**适用场景**：任何需要从多轮对话中提取特定回复的自动化场景。

### 模式 2: 自举配置（Bootstrap Configuration）

**问题**：skill 需要适配不同的本地环境（shell、Python、浏览器），但不能硬编码。

**方案**：
1. 首次运行时探测环境（运行命令 → 解析输出）
2. 写入 JSON 配置文件
3. 后续运行直接读取，不重复探测
4. 失败时提示用户"环境可能变更，是否重新配置？"

**适用场景**：任何跨平台的 CLI 工具或 skill。

### 模式 3: 两阶段代码生成（Two-Phase Code Generation）

**问题**：不同 AI 平台的 Web 界面 DOM 结构不同，无法写一套通用代码。

**方案**：
1. **定型阶段**：LLM 分析页面 DOM → 生成平台专属的交互脚本 → 缓存
2. **使用阶段**：加载缓存的脚本直接执行，不重复分析 DOM

**适用场景**：任何需要适配多个 Web 平台的自动化 agent。

### 模式 4: 异常恢复优先（Fail-Fast + Retry）

**问题**：复杂的多步骤流程容易在中间状态卡住，debug 成本高。

**方案**：
1. 每个步骤失败时，不修复中间状态
2. 直接从头重跑整个流程
3. 流程设计上保证重跑不会产生副作用（idempotent）

**适用场景**：任何多步骤 agent 工作流。

## 5. 下一步：代码 Agent 优化方向

当前 skill 面向"通用搜索"，下一阶段针对**代码编写辅助**优化：

### 待优化点
- **搜索 prompt 模板**：增加代码上下文（语言、框架、文件路径）的注入
- **提取后的后处理**：自动提取代码块、diff、API 签名等结构化信息
- **与编辑器集成**：提取结果直接填充到代码补全、注释、文档生成

- **减少延迟**：搜索 → 提取的等待时间对代码编写场景仍然太长

## 6. v6 架构变更（2026-05）

### ADR-6: 提取器不再做格式验证

**问题**：v5 的 `is_content_complete` 硬编码检查 `["概述","核心要点","来源"]`，debug 模式输出格式（根因分析/证据/候选方案）完全不匹配，导致提取永远被拒绝。

**决策**：`is_content_complete` 只做存在性检查（非空 + >=150 字符 + 非 CoT + 非 prompt）。格式正确性是 prompt 设计的事，提取器不越界验证。

**影响**：新增搜索模式不再需要修改提取器代码。

### ADR-7: 标记对直接定位，不再依赖 rfind(prompt)

**问题**：v5 用 `rfind(prompt文本)` 在页面中定位发送位置。多轮对话中 prompt 文本可能被 AI 引用，rfind 找错位置导致提取到旧内容。

**决策**：纯标记对定位——收集所有 `[搜索主题：xxx@hash]` 位置 → 取最后一对 → 提取中间内容。@hash 保证全局唯一，不需要额外定位锚点。

### ADR-8: CoT 降级策略

**问题**：DeepSeek 的思考过程（CoT）在 innerText 中可见，标记对可能出现在 CoT 中导致误提取到思考过程而非正式回复。

**决策**：提取后检测 CoT 特征（头部含"已思考"/"搜索到"，尾部含"内容由 AI 生成"）。检测到 CoT → 取最后一个标记之后的内容作为正式回复。

### ADR-9: 轮询参数收紧

**问题**：v5 的 5s 轮询 + 2 次稳定确认 = 至少 10s 确认等待，对快速回复的 AI 来说过长。

**决策**：2s 轮询 + 3 次稳定确认 = 6s 确认等待。CoT 检测降低了误判风险，可以更激进。

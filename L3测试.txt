概述
从零构建企业级 LLM 推理平台涉及推理引擎选型、编排调度、网关与可观测性三大核心决策轴心。根据 2026 年 Q1 公开基准数据，vLLM 在 H100 集群上处理 128 并发时可达 4300 tokens/sec，TensorRT-LLM 可达 6200 tokens/sec 的峰值吞吐，但后者需要约 12 小时的模型编译与配置时间，而 vLLM 可在 1 小时内完成初始部署。TGI 已于 2025 年 12 月进入维护模式，Hugging Face 官方不再推荐用于新建项目。Ollama 在企业级高并发场景下存在显著性能瓶颈，当请求量超过 5 QPS 或模型参数量突破 130 亿时，会出现明显的性能衰减。以下从七个核心技术决策点展开分析。

核心要点
1. 推理引擎选型：vLLM 为生产标准，TensorRT-LLM 为极致性能备选

vLLM 凭借 PagedAttention 内存管理技术，将 KV Cache 内存碎片率从传统方案的 35% 以上降至 4% 以下，有效显存利用率提升 5.8 倍。在 NVIDIA RTX 5090 上运行 LLaMA 3.1 70B（4-bit 量化）时，vLLM 0.8.x 吞吐达 68 tokens/sec，首 Token 延迟 125 ms；TensorRT-LLM 达 78 tokens/sec，首 Token 延迟 95 ms。在 Kubernetes + 2×H100 生产环境中，128 并发下 vLLM 吞吐 4300 tokens/sec，Triton+TRT-LLM 达 6200 tokens/sec。vLLM 的 V1 引擎自 v0.7.0 起成为默认，Anyscale、IBM、Databricks、Cloudflare 等头部公司已在其生产环境采用。

2. 推理引擎备选：SGLang 在多轮对话场景表现突出

SGLang 通过 RadixAttention 实现前缀树级 KV 缓存共享，在多轮对话、Agent 等具有共享上下文的场景中，缓存命中率可达 85%–95%。在 H100 上的批量推理 benchmark 中，SGLang 以 16200 tokens/sec 超越 vLLM 的 12500 tokens/sec（约 29% 领先）。对于 Agent 应用和结构化输出场景，SGLang 是优于 vLLM 的选择。

3. 轻量级工具的明确边界：Ollama 不适合生产高并发

Ollama 定位于开发和原型验证，不支持张量并行，在 5 QPS 以上即出现明显排队。RTX 5090 上 LLaMA 3.1 70B 测试中，Ollama 仅达 38 tokens/sec，首 Token 延迟 210 ms，约为 vLLM 性能的 56%。某智能客服团队实测 Ollama 部署 7B 模型时 99 分位延迟达 3.2 秒，无法满足商业级 SLA 要求。

生产级 LLM 推理平台需采用三层架构：计算资源层（Kubernetes GPU 集群）、推理服务层（vLLM/TensorRT-LLM 实例）、流量控制层（AI 网关）。Kubernetes 集群中，通过 NVIDIA Device Plugin 实现 GPU 资源统一抽象，结合 HPA（Horizontal Pod Autoscaler）与 VPA（Vertical Pod Autoscaler）的联合调度策略，可使资源利用率提升至 65% 以上。结合 AI 网关的故障自动切换机制，服务可用性可提升至 99.99%，故障切换时间缩短至 800 ms 以内。

6. AI 网关与多模型统一管理

企业级推理平台需提供 OpenAI 兼容的统一 API 接口，支持 vLLM、SGLang、TensorRT-LLM 等多引擎的聚合接入。托管型网关方案在压力测试中可支撑每秒 2 万次调用，P99 延迟稳定在 500 ms 以内，故障迁移延迟低于 0.5 秒；自建型方案在并发超过 3000 时即触发限流，约 20% 的请求被丢弃。Kong AI Gateway 与 LiteLLM 在企业级流量治理、成本控制与多模型路由方面各有侧重。

风险评估
坑点 1：TGI 进入维护模式，新建项目应避免选型

Hugging Face TGI 已于 2025 年 12 月进入维护模式，官方推荐 vLLM 或 SGLang 用于新建部署。若现有生产环境使用 TGI，建议规划 3–6 个月的迁移窗口。

坑点 2：Ollama 被误用于生产高并发场景

Ollama 在 5 QPS 以上即出现显著排队，不支持张量并行，70B 模型吞吐仅 38 tok/s（vLLM 的 56%）。某电商团队在促销期间因此出现持续 4 小时服务中断。明确将 Ollama 限定在开发/原型阶段。

坑点 3：vLLM 在多卡部署中的通信开销

vLLM 支持张量并行与流水线并行，但跨 GPU 通信（NCCL）会引入额外延迟。对于 70B 以上模型，建议优先使用 NVLink 互联的 GPU（如 2×A100 80GB），避免跨节点通信带来的性能损耗。

坑点 4：TensorRT-LLM 模型编译的版本与硬件约束

TensorRT-LLM 需要将 HuggingFace 格式模型编译为 NVIDIA 专有引擎格式，此过程耗时约 12 小时，且引擎与特定 GPU 架构（如 H100 的 Hopper 架构）强绑定。更换硬件或升级驱动版本后需重新编译，严重限制模型迭代速度。

坑点 5：自建 AI 网关的并发瓶颈

某企业自建中转层在并发数超过 3000 时触发限流，约 20% 的请求被丢弃。建议在生产环境优先评估托管型网关方案，或基于 Kong/LiteLLM 等成熟开源网关构建，避免从零实现负载均衡与熔断逻辑。

坑点 6：GPU 显存碎片化导致资源浪费

传统推理框架在处理每秒数百请求时，GPU 内存碎片化会导致 30% 以上的算力浪费。vLLM 的 PagedAttention 可将碎片率控制在 4% 以下，但需正确配置 block_size 参数（推荐 16 或 32），不当配置可能导致缓存命中率下降。

坑点 7：开源方案在生产环境的安全合规风险

部分开源推理框架默认不启用传输加密和访问控制，可能违反等保 2.0 三级要求。需在外层配置 mTLS、JWT 验证与 WAF 模块，并在网关层实现 API 鉴权与审计日志。
